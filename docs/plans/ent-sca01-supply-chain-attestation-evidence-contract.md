# ENT-SCA01 Supply Chain Attestation Evidence Contract

> Status: Input document. This contract defines the evidence required for release
> artifact attestation. It does not claim attestation is implemented, authorize
> production operations, or replace the active program/tracker authority.

## Purpose

Move the supply-chain attestation enterprise-readiness lane from "not yet scoped" to a
concrete evidence contract. A future implementation slice can use this contract to prove that
release artifacts are built, identified, signed, attested, and verified before promotion.

## Current Repo Evidence

Existing release evidence is strong for repo delivery, but incomplete for supply-chain
attestation:

- `.github/workflows/cd.yml` builds and pushes GHCR images tagged with the release commit SHA for
  staging and production.
- Deployment verification checks `/api/health` and fails when the deployed `build.commitSha` does
  not match the expected GitHub SHA.
- Staging and production release gates upload bounded verification artifacts.
- Required repo checks include CodeQL, SonarCloud, gitleaks, pnpm audit, `pnpm security:guard`,
  PR finalization, and release-gate verification.

The repo does not currently prove:

- per-release SBOM generation for the deployed image;
- signed build provenance bound to repository, workflow, ref, commit SHA, and image digest;
- image or artifact signature creation by a trusted CI identity;
- deploy-time signature, attestation, and SBOM verification; or
- deployed artifact digest equality against the attested digest.

## Scope

This contract covers SBOM identity and hash, immutable image digest custody, build provenance,
artifact signature or attestation identity, deploy-time verification, deployed digest equality, and
pass/fail blocker capture for GHCR release artifacts promoted by the repo CD workflow.

Out of scope:

- Production deployment execution.
- Credential, environment, registry, or deploy-webhook changes.
- Runtime code, schema, RLS, auth, tenancy, routing, billing, product UI, proxy, README, AGENTS, or
  broad architecture-document changes.
- Simulated signing, local-only attestations, or unverifiable operator assertions.

## Acceptance Criteria

An attestation record satisfies this contract only when:

- the deployed image digest is uniquely identified and does not rely on a mutable tag alone;
- an SBOM is generated for the same digest and recorded with an artifact identity and hash;
- provenance binds the artifact digest to the repository, workflow, ref, and commit SHA;
- the signature or attestation is created by a trusted CI identity;
- deployment or promotion verifies the signature, provenance, SBOM binding, and deployed digest;
- deployed digest verification proves equality with the attested digest;
- the record contains no secrets, credentials, raw PII, claim narratives, document contents, or
  payment data;
- a named owner records a pass/fail decision; and
- any blocker has a follow-up issue, PR, or explicit risk owner.

## Recommended Next Implementation

The next unblocked implementation slice should add a CI or CD proof that generates an SBOM,
captures the immutable image digest, emits signed provenance or artifact attestation from the
trusted GitHub workflow identity, and verifies those materials before deploy promotion. If the
registry, runner, or environment lacks attestation permissions, record that provider blocker
instead of claiming success. This contract scopes one enterprise maturity requirement; it does not
by itself prove full enterprise readiness.
